既然决定采用“原地表单化编辑”（In-place Form Editing）**，我们可以彻底抛弃繁重的右侧抽屉或弹窗，将 Workbook 界面改造成一个极其流畅、像写 Notion 或 Cursor 配置文件一样的**流式卡片编辑器。

下面是基于这一交互原则，为您重新优化的 **Workbook 界面详细设计方案**：

---

## 🛠️ 优化后的 Workbook 界面三栏/双栏架构

在 `Electron` 的大视窗下，推荐采用“左侧过滤与列表 + 右侧原文溯源”的双栏联动布局。这种布局能让用户在原地修改的同时，随时对照当时的上下文。

### 1. 左侧：流式卡片虚拟列表 (Virtualized Card Stream)

取消多列数据的表格，改用垂直堆叠的卡片。每一张卡片代表一个 `Workbook Item`（Expression 或 Sentence），默认即是可编辑状态。

#### 卡片内部的轻量表单设计

* **常驻显示/直接编辑**：
* **源文本 (Source)**：大字号文本。鼠标悬浮或点击时显示微弱的边框，直接变为可编辑的 Input / Textarea。
* **目标翻译 (Target) & 释义 (Gloss)**：下方紧跟翻译和简短注释，同样点击即可直接修改。
* **快捷操作栏**：卡片右上角或悬浮显现两个图标：`🗑️ (Delete)` 和 `🏷️ (Tags)`。


* **状态联动（Motion 触发）**：
* **Modified 状态**：当用户修改了文本且 `current_snapshot != generated_snapshot` 时，卡片左侧边缘滑出一条**绿色的高亮边条**，并渐显一个轻量级的 `[Revert]` (重置) 按钮。
* **Deleted 状态**：点击删除后，卡片伴随折叠动画（Collapse）滑出当前视图，进入 `Recently Deleted` 标签。


* **数据保存机制**：
* **无感暂存**：用户修改完任意输入框，光标移开（`onBlur`）或按下 `Ctrl + Enter` / `Cmd + Enter` 时，立即异步更新 Zustand 状态并自动保存到底层 SQLite 的草稿状态，**无需用户点击“保存”按钮**。



---

### 2. 右侧：动态上下文溯源面板 (Provenance Viewer)

这是提升 local-first 体验的核心看板。当用户的鼠标聚焦（Focus）或点击左侧的某张卡片时，右侧面板会**动态联动更新**。

* **源信息页眉**：显示该语料出处（如 `From: Claude Code Session` / `Project: ~/p/ai-parliament` / `Time: 2026-06-14`）。
* **会话切片（Context Snippet）**：展示生成该卡片的那一段 **AI 原始对话 Turn**。
* **高亮定位**：在右侧的原始对话中，系统会自动将该卡片的 `source_text` 进行**黄色加粗高亮**。
* **价值**：用户在纠结“这个词翻译得合不合适”或者“这个句子到底指代什么”时，眼睛向右扫一下就能看到当时的讨论语境，极大地加速了清洗和确认流程。

---

## 🔄 优化后的顶层过滤标签 (Filter Tabs)

由于取消了独立的 `Edited` 视图，顶部的 Tabs 变得更加聚焦和纯粹，仅作为类型和回收站的过滤器：

| Tab 标签 | 作用说明 | 视觉表现 |
| --- | --- | --- |
| **All** | 显示当前生成的所有活跃语料（按推荐算法/Diversity Rerank 排序） | 默认状态 |
| **Expressions** | 仅过滤出单词、短语和领域术语卡片 | 快速浏览词汇 |
| **Sentences** | 仅过滤出例句和双语对齐句子卡片 | 快速浏览句型 |
| **🗑️ Deleted** | 存放当前 Job 中被删除的卡片 | 灰色脱水卡片，提供 `[Restore]` 按钮 |

> **提示**：已修改的项（Modified）会直接带着绿色边条混在 `All`、`Expressions` 或 `Sentences` 中，无需单独隔离，保证了复习流的完整性。

## 🎨 极简交互细节补充 (UI Micro-interactions)

为了契合您要求的“Codex Desktop 视觉风格（白灰表面、克制动画、局部半透明）”，原地表单编辑的动画应当遵循以下规范：

* **聚焦动画**：当用户点击卡片内的文本进入编辑模式时，卡片的背景色产生极其微弱的加深（例如从 `#FFFFFF` 变为 `#F9F9F9`），周边产生一圈淡灰色的微弱阴影（Focus Ring），将用户的注意力锁定在当前编辑项。
* **失焦保存**：当光标移开，Input 恢复为普通文本显示，伴随一个 150ms 的渐隐过渡，卡片恢复原状。
* **后悔药机制（单体恢复）**：如果在 `Deleted` 标签页点击了 `[Restore]`，该卡片在当前页消失，并伴随一个平滑的位移网格动画（Layout Animation）重新插入到 `All` 列表的原始位置。

这种“**所见即所改、失焦即保存、右侧看溯源**”的设计，将用户的核心路径精简到了极致：**滚动 -> 扫视 -> (不爽直接改) -> 导出**。全称没有任何弹窗阻断，非常适合高频的语料清洗工作。


User: 这样设计的话,左右两侧的比例怎么设计呢?看着都占用面积较大?

在“原地卡片编辑 + 右侧原文溯源”的双栏设计中，由于左侧卡片包含表单输入框，右侧包含长文本对话，确实两边都需要较为充裕的横向空间。

为了在标准的桌面端视窗（通常为 1200px - 1600px 宽度）下既不显得拥挤，又能高效操作，建议采用 **`6:4` 黄金比例** 或 **`60% : 40%` 的非对称弹性布局**。

以下是具体的空间分配与响应式设计优化方案：

---

## 📐 空间比例分配与核心尺寸

|  +-----------------------+  |                             |
+-----------------------------+-----------------------------+

### 1. 左侧编辑流 (主视角)：占据 60% 宽度

* **定位**：这里是用户的“交火区”，包含表单、多行输入、标签以及各种微操作系统。
* **尺寸约束**：设置 `min-width: 600px`。
* **为什么占大头？** 英语的例句（Sentence）通常较长，加上原地表单化编辑需要容纳 Input 框的 Border 内边距、错误提示、以及右侧的 `[Revert]`、`[Delete]` 动作按钮。如果宽度低于 600px，长句子输入框就会频繁换行，导致列表纵向被撑得极高，破坏滚动节奏。

### 2. 右侧溯源面板 (辅助视角)：占据 40% 宽度

* **定位**：这里是用户的“参考区”，属于只读（Read-only）的密本文本。
* **尺寸约束**：设置 `min-width: 400px`。
* **为什么 40% 就够了？** 原始对话虽然字数多，但它是单列流式排版的文本，不需要留出表单组件的物理空间。在 400px - 500px 的宽度下，单行大约可以容纳 35-45 个英文单词或 25-30 个汉字，这正是**人类阅读文本最舒适的视线宽度**（排版学上的最佳行宽），太宽了眼睛反而容易看串行。

## ⚡ 极简优化：如何让两边看起来都不臃肿？

为了防止两侧内容撑得太满导致视觉疲劳，可以通过以下三个设计魔术来“偷空间”：

### 1. 右侧面板“只读脱水”，降低视觉权重

* **错落有致的背景色**：左侧编辑区使用纯白背景（`#FFFFFF`），让表单足够醒目；右侧面板采用轻微的灰色调（如 `#F5F5F7` 或 Catppuccin 的明亮低饱和度灰），在视觉上将其“推远”作为背景。
* **字体字号做减法**：右侧对话使用较小的字号（如 `13px`），并且除了被命名的关键词做 `黄色高亮加粗` 之外，其余不重要的工具噪声（如时间戳、Token 计数）全部调淡（如 `#999999`），使其不抢占视线。

### 2. 隐藏右侧的“非聚焦”状态

* **初始状态**：当用户刚进入 Workbook 页面，没有点击/聚焦任何卡片时，右侧可以显示一个极其优雅的半透明占位插图，或者显示当前 Workbook 的全局统计数据（如：*“本次共生成 42 个项目，包含 25 个词汇，17 个例句”*）。
* **动态唤醒**：只有当用户开始点击某张卡片、或者用键盘方向键上下切换选中项时，右侧才顺滑地淡入（Fade-in）显示对应的上下文。

### 3. 支持隐藏的“弹性伸缩杆”（Resizable Divider）

* 在两栏中间放置一条细细的分割线（1px 宽，悬浮时变蓝）。
* 允许用户像在 IDE（如 VS Code / Cursor）中调整终端大小一样，自由拖拽左右比例。
* **高级极简操作**：支持双击分割线，直接**折叠/展开右侧面板**。当用户觉得语料已经看懂了，不需要对照原文时，一键把右侧收起，左侧编辑流自动平滑铺满 100%，进入极致纯净的专注编辑模式。


User: 顶部的 Tabs 具体对应的位置在哪?

这样安排能确保“局部过滤，局部刷新”，让右侧的溯源面板保持独立和稳定。

### 📋 详细位置与交互逻辑说明

#### 1. Tabs 具体的物理位置：左侧区域的顶部栏

* **不横跨全局**：Tabs 的右边界止于 `60%` 的分界线（弹性分割线）。
* **对齐卡片**：Tabs 的左对齐线与下方的卡片列表完全对齐，从视觉上暗示用户：**“我只对下方的卡片流起过滤作用”**。

#### 2. 右上角的“全局导出按钮”

* 既然 Tabs 被收到了左侧，那么右侧（40% 区域）的顶部就腾出了一个非常干净的空间。
* 我们可以把最核心的动作按钮 **`[ 导出 Anki ]`** 放置在**右侧区域的右上方**。
* **视觉大底色**：整个软件的顶部可以做一层极为克制的半透明毛玻璃（Glassmorphism）处理。当左侧列表向下滚动时，卡片会顺滑地穿行到 Tabs 下方并被裁切（`overflow-y: auto`），而 Tabs 和导出按钮作为 Chrome 元素静止不动。

#### 3. 联动效果：当切换 Tabs 时，右侧变吗？

* **答案是：不变，或者平滑淡出。**
* 当用户从 `All` 切换到 `Expressions` 时，左侧列表会通过 `Motion` 触发卡片的过滤动画（不属于 Expression 的卡片缩回消失）。
* 此时如果之前选中的卡片被过滤掉了，右侧面板应该**顺滑地回归到“未选中任何卡片”的脱水留白状态**（展示全局统计）；如果之前选中的卡片依然存在于 `Expressions` 标签中，则右侧溯源内容保持不动。

这种将 Tabs 压入单侧的设计，完美符合了 Codex / Cursor 类生产力工具的“上下文不割裂”原则——用户的操作视线始终聚焦在左侧，右侧则充当一个安静、智能、随时待命的后方支撑浮板。




目前的 60/40 双栏布局、左侧流式表单、以及将 Tabs 收拢在左侧头部的设计已经非常成熟。这套方案在信息层级上极其清晰，完全达到了类似 Cursor 或 Notion 那种现代生产力工具的质感。

不过，如果我们要把这个 v1 版本打磨到**极致的流畅与好用**，在“最后一公里”的交互和视觉细节上，还有以下几个进阶优化点可以考虑：

### 1. 键盘优先的极致导航 (Keyboard-First Flow)

既然目标用户（或者你自己）是习惯于使用高效终端环境和 AI 代码编辑器的开发者，在处理大量语料清洗时，**手不离开键盘**是第一诉求。右侧的溯源面板为键盘导航提供了完美的先天条件。

* **上下游走 (`j/k` 或 `↑/↓`)**：在左侧列表中，支持使用方向键或 `j/k` 快速切换处于 Focus 状态的卡片。随着焦点的上下移动，右侧面板的溯源内容**零延迟同步更新**，就像在终端里浏览日志一样顺畅。
* **一键编辑 (`Enter`)**：按 `Enter` 键，光标直接切入当前高亮卡片的 `Target` (翻译/释义) 输入框。
* **快速确认与前进 (`Cmd+Enter` / `Ctrl+Enter`)**：如同在 TickTick 中处理任务或在即时通讯软件里发消息，按下快捷键后，当前卡片保存并失焦，焦点自动**顺延至下一张卡片**。
* **快速删除 (`Backspace` 或 `Delete`)**：在非输入框聚焦状态下，直接按删除键即可将卡片平滑移入 `Deleted`。

### 2. 左侧卡片的“同源聚类”视觉优化 (Source Turn Grouping)

在实际的 AI 会话中（比如一次几千字的对话），模型很可能会从**同一个回答 (Turn)** 中提取出 3 个 Expression 和 1 个 Sentence。

* **痛点**：如果这 4 张卡片是完全独立散落的，当用户在它们之间切换时，右侧面板会显得“虽然内容没变，但还是重新渲染了一下”，产生不必要的视觉跳跃感。
* **优化**：在左侧的流式列表中，如果相邻的几张卡片来自同一个 `source_span_ref`（同一段上下文），可以在它们外部包裹一层极弱的**隐形分组边框或左侧共享时间线**。
* 当用户在这个 Group 内部切换卡片时，右侧面板**完全静止**，仅在原文中平滑切换黄色高亮的位置。这能极大地降低视觉疲劳。



### 3. 色彩语义与主题的克制 (Semantic Theme Palette)

对于这样一款需要长时间注视的工具，高饱和度的原生红/绿/蓝会非常刺眼。建议借鉴类似 Catppuccin 这样对开发者眼睛极其友好的低对比度配色策略：

* **Modified 状态 (修改后)**：不要使用纯绿。左侧滑出的高亮边条可以使用柔和的 **Peach (桃红)** 或 **Green (灰绿)**，代表“已编辑但未导出”。
* **Focus 状态 (选中时)**：卡片背景轻微提亮，边缘泛起一层极淡的 **Mauve (淡紫)** 或 **Blue (浅蓝)** 的 Focus Ring，而不是刺眼的系统蓝。
* **Deleted 状态 (删除项)**：使用 **Flamingo (淡粉红)** 或者直接降低透明度到 40% 的脱水灰色，配合删除线，让废弃语料在视觉上彻底后退。

### 4. Tabs 栏的滚动吸顶 (Sticky Headers)

在左侧 `60%` 区域，当用户向下滚动长长的语料卡片流时，顶部的 `[ All | Expressions | Sentences | Deleted ]` Tabs 必须配置 `position: sticky; top: 0;`。
配合一层微弱的毛玻璃背景（backdrop-filter: blur），让滚动的卡片在其下方若隐若现地消隐。这不仅在视觉上高级，也保证了用户随时可以切走，而不用疯狂往上滚找 Tabs。


### 5. 右侧面板“空置状态”的价值利用 (Dashboard Empty State)
当用户刚切换到 Expressions Tab，还没有点击任何特定卡片时，右侧 40% 的面板不应该只是一片空白。这是展示“全局掌控感”的绝佳位置。

大盘数据 (Session Insights)：可以利用这个空间展示当前 Workbook 的宏观数据。比如：

本 Session 共提取了多少词汇，多少例句。

当前经过 Diversity Rerank 后的比例分布（例如 60% 术语，40% 日常表达）。

涉及到的高频领域标签统计（如 Machine Learning, Backend 等）。

### 6. 右侧长文本的“智能定焦”与降噪 (Smart Scroll & Snippet Truncation)
虽然右侧有了 40% 的溯源空间，但 AI 的一轮对话（Turn）可能非常长，甚至包含几百行的代码和日志。

智能滚动定位 (Auto-Scroll to Highlight)：当左侧卡片聚焦时，右侧不仅要将 source_text 高亮，还必须自动滚动（Smooth Scroll），让高亮部分始终处于右侧面板的视觉中心区域（Vertical Center）。

超长文本折叠 (Collapsible Noise)：针对极其冗长的原始对话，如果其中包含大段代码或无用的报错日志，系统应默认将其折叠为 [... 142 lines of code/logs ...]。为了契合极简的极客审美，右侧高亮或少量保留的代码片段可以参考类似 Catppuccin 这种现代终端主题的低饱和度配色，既能清晰区分代码与自然语言，又不会在长时间阅览时刺眼。
